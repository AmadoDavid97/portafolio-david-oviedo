Optimización de Spring Boot para Producción: Guía Completa

Amado David Oviedo Quintana
15 Oct 2024 8 min de lectura

Después de 3 años desarrollando y optimizando eVendeha, nuestro sistema comercial integral construido con Spring Boot, he aprendido que llevar una aplicación de desarrollo a producción exitosamente requiere mucho más que código funcional. En este artículo, comparto las técnicas y optimizaciones que nos han permitido mantener un sistema escalable, rápido y confiable.

¿Por qué es importante la optimización?

En eVendeha, manejamos operaciones críticas como ventas en tiempo real, facturación electrónica y gestión de inventarios para múltiples clientes simultáneamente. Una aplicación mal optimizada puede resultar en:

  • Tiempos de respuesta lentos que frustran a los usuarios
  • Alto consumo de recursos que incrementa costos de hosting
  • Problemas de escalabilidad cuando crece la base de usuarios
  • Caídas del sistema en momentos de alta demanda

1. Configuración Óptima de la JVM

La configuración de la JVM es fundamental para el rendimiento. Aquí están los parámetros que utilizamos en eVendeha:

application.properties 
# Configuración JVM recomendada para producción
java -Xms2G -Xmx2G \
  -XX:+UseG1GC \
  -XX:MaxGCPauseMillis=200 \
  -XX:+HeapDumpOnOutOfMemoryError \
  -XX:HeapDumpPath=/var/log/heapdump.hprof \
  -Dspring.profiles.active=prod \
  -jar eVendeha.jar

Explicación de los parámetros:

  • -Xms2G -Xmx2G: Establecer el mismo tamaño inicial y máximo evita el redimensionamiento dinámico del heap
  • -XX:+UseG1GC: El recolector G1 ofrece mejor rendimiento para aplicaciones con heaps grandes
  • -XX:MaxGCPauseMillis=200: Limita las pausas de GC a 200ms para mantener la aplicación responsive
  • -XX:+HeapDumpOnOutOfMemoryError: Genera un dump cuando hay OutOfMemory para debugging

2. Optimización de Conexiones a Base de Datos

El pool de conexiones es crítico. Una configuración incorrecta puede saturar la base de datos o desperdiciar recursos.

application-prod.yml 
spring:
  datasource:
    hikari:
      maximum-pool-size: 10
      minimum-idle: 5
      connection-timeout: 20000
      idle-timeout: 300000
      max-lifetime: 1200000
      leak-detection-threshold: 60000
  jpa:
    hibernate:
      ddl-auto: validate
    properties:
      hibernate:
        jdbc:
          batch_size: 20
        order_inserts: true
        order_updates: true
        query:
          in_clause_parameter_padding: true
Consejo Pro: En eVendeha calculamos el pool size óptimo con la fórmula: connections = ((core_count * 2) + effective_spindle_count). Para un servidor con 4 cores y un SSD (spindle=1), usamos 10 conexiones.

3. Caching Estratégico

Implementamos caching en múltiples niveles para reducir la carga en la base de datos y mejorar los tiempos de respuesta.

CacheConfig.java 
@Configuration
@EnableCaching
public class CacheConfig {
    
    @Bean
    public CacheManager cacheManager() {
        CaffeineCacheManager cacheManager = new CaffeineCacheManager();
        cacheManager.setCaffeine(Caffeine.newBuilder()
            .maximumSize(1000)
            .expireAfterWrite(10, TimeUnit.MINUTES)
            .recordStats());
        return cacheManager;
    }
}

@Service
public class ProductService {
    
    @Cacheable(value = "products", key = "#id")
    public Product findById(Long id) {
        return productRepository.findById(id)
            .orElseThrow(() -> new NotFoundException("Producto no encontrado"));
    }
    
    @CacheEvict(value = "products", key = "#product.id")
    public Product update(Product product) {
        return productRepository.save(product);
    }
}

4. Lazy Loading y Paginación

Nunca cargar toda la data de una vez. En eVendeha, todas las listas implementan paginación:

ProductController.java 
@GetMapping("/products")
public ResponseEntity> getProducts(
    @RequestParam(defaultValue = "0") int page,
    @RequestParam(defaultValue = "20") int size,
    @RequestParam(defaultValue = "id,desc") String[] sort
) {
    Pageable pageable = PageRequest.of(page, size, 
        Sort.by(Sort.Direction.fromString(sort[1]), sort[0]));
    
    Page products = productService.findAll(pageable);
    return ResponseEntity.ok(products);
}

5. Compresión de Respuestas HTTP

Habilitar compresión GZIP reduce significativamente el tamaño de las respuestas:

application-prod.yml 
server:
  compression:
    enabled: true
    mime-types:
      - application/json
      - application/xml
      - text/html
      - text/xml
      - text/plain
    min-response-size: 1024

6. Monitoreo con Actuator

Spring Boot Actuator nos permite monitorear la salud y métricas de la aplicación en tiempo real:

pom.xml 

    org.springframework.boot
    spring-boot-starter-actuator


    io.micrometer
    micrometer-registry-prometheus
application-prod.yml 
management:
  endpoints:
    web:
      exposure:
        include: health,metrics,prometheus
  endpoint:
    health:
      show-details: when-authorized
  metrics:
    export:
      prometheus:
        enabled: true

7. Async Processing

Para operaciones que no requieren respuesta inmediata, utilizamos procesamiento asíncrono:

AsyncConfig.java 
@Configuration
@EnableAsync
public class AsyncConfig {
    
    @Bean(name = "taskExecutor")
    public Executor taskExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(5);
        executor.setMaxPoolSize(10);
        executor.setQueueCapacity(100);
        executor.setThreadNamePrefix("async-");
        executor.initialize();
        return executor;
    }
}

@Service
public class EmailService {
    
    @Async("taskExecutor")
    public CompletableFuture sendInvoiceEmail(Invoice invoice) {
        // Enviar email sin bloquear el hilo principal
        emailSender.send(invoice);
        return CompletableFuture.completedFuture(null);
    }
}

Conclusiones

La optimización de aplicaciones Spring Boot no es una tarea única, sino un proceso continuo. Las técnicas presentadas aquí son las que mejor han funcionado para eVendeha, pero cada aplicación tiene sus propias necesidades.

Recomendaciones finales:

  1. Siempre mide antes de optimizar (usa profilers como JProfiler o YourKit)
  2. Optimiza los cuellos de botella reales, no los percibidos
  3. Implementa monitoreo desde el día uno
  4. Documenta cada cambio de configuración
  5. Prueba las optimizaciones en un ambiente similar a producción
